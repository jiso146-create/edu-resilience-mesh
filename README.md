超高密度メッシュ型 浸水監視システム(仮称)

低コスト・高耐久・自律メンテナンスフリーを目指した、教育目的の浸水監視システムです。すべての設計・回路・ファームウェアを MIT ライセンスのもとオープンソースとして公開しています。

⚠️ はじめに読んでください これは学校の学習プロジェクトです。地域のみなさんには「参考情報」としてご覧いただくことを想定しています。実際の避難の判断は、必ず自治体の防災情報および気象庁の情報を最優先してください。 本システムの表示や通知は、公式の避難情報に優先するものでは一切ありません。

このプロジェクトの設計思想
1. 「めったに動かないこと」を正常とする
このシステムは、大半の時間、何も検知しません。それが正常な状態です。側溝や通学路は、一年のほとんどを乾いた状態で過ごします。私たちが挑戦したのは、「めったに起きない異常」を、いかに低コストで、電池交換なしで、壊れずに、ひたすら待ち構え続けるか、という一点です。防災とは「99.9%の空振り」を続けられるかどうかの戦いだと、私たちは考えました。

2. 引き算の設計 ― 守るものを絞り、守らないものを決める
すべてを守ろうとすると、高価で壊れやすくなります。そこで私たちは「どこまで守り、どこから先は諦めるか」を最初に線引きしました。水位は連続値では測らず、5cm(注意)・15cm(危険) の二段階だけを電極の高さの差で検知します。それを超える冠水では、デバイス自身が水没して応答を止めます。この**「応答なし(ロスト)」という状態そのものを、最大の危険信号(黒)として扱います。** センサが壊れて黙ることが、最後の、そして最も雄弁な通信になる、という設計です。

3. 普段は寝て、水が来たら起きる
省電力の答えは「普段は寝ていること」でした。電極は平常時には通電せず、消費電力はほぼゼロです。マイコンは 2時間おきにハートビート信号(生存確認) だけを送って、また眠ります。この間隔は、電力の節約・ロスト検知の速さ・通信の混雑という三つのバランスを考えて決めました。

4. 二段構えの監視モード
水位が 5cm に達した子機は、自ら観測間隔を約10分に切り替えます(自律格上げ)。さらに、大雨警報などの状況に応じて、中央のサーバ(GAS)側から全子機へ「観測頻度を上げよ」という指令を一斉に下ろすことができます(中央格上げ)。デバイス自身の判断と、上位からの判断。この二つで監視の目を細かくします。ただし、省電力のために子機が眠っている間は指令がすぐ届かないため、指令の反映には最大でハートビート間隔ぶんの遅れが生じます。この「即応性と省電力のトレードオフ」は、私たちが今も向き合っている設計課題です。

5. 通信のランニングコストを「構造」で消す
多数のセンサを長く運用するとき、最大の敵は毎年かかる通信費です。私たちは携帯回線(LPWA / LTE)を使わず、子機どうしが Thread(IPv6 メッシュ) で互いに中継し合い、親機(Border Router)が学校や公民館の既存 Wi-Fi に接続します。バックエンドは Google Apps Script(GAS) の無料枠を使います。こうして、通信の維持費を構造的にほぼゼロにしました。1台が失われても、周囲の子機が別の中継ルートを自動で組み直します(自己修復)。

6. 土地に合わせて最適化する ― 晴れの国だから成立する電源
電源は太陽光パネルとスーパーキャパシタ(電気二重層コンデンサ)のみで構成し、化学二次電池を使いません。液漏れ・発火のリスクがなく、超高サイクル寿命で、高温・低温にも強いためです。この構成は日照に依存しますが、私たちの住む倉敷は「晴れの国」岡山。年間を通じて日照に恵まれるこの土地だからこそ、小さなパネルと蓄電で成立します。エンジニアリングの最適解は、その土地の環境で変わる、ということを学びました。

7. マイコンは消耗品、規格は資産
私たちは特定のチップに心中しません。通信に業界標準の Thread / 802.15.4 を採用したことで、将来より優れたマイコンが登場すれば、ネットワークはそのままに子機だけを載せ替えられます。オープンソースと標準規格は、この「未来への乗り換え可能性」を守るためにあります。

既存の取り組みとの関係(学習上の位置づけ)
同じ課題には、国土交通省が「ワンコイン浸水センサ実証実験」として全国規模で取り組んでいます(令和7年度は232自治体が参加)。私たちのプロジェクトは、それと競うものでも、それに代わるものでもありません。「同じ課題に、国はこう答えた。では、教材の私たちはどう答えられるか」 を考えるための学習であり、公的な取り組みと個人・地域の工夫は、速さ・網羅性・責任の所在という点でそれぞれ役割が違う、ということを学ぶ題材です。
